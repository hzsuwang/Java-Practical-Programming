# 并发编程实战之三：线程池的7个致命参数，90%的线上故障源于此

## 一、场景引入：凌晨2点的告警电话

凌晨2点15分，我被运维的告警电话吵醒。

"订单系统接口响应超时率飙升到40%，CPU使用率正常，但GC频繁Full GC！"

我打开监控一看——堆内存已经用到95%，老年代持续增长。jstack一看，瞬间明白问题所在：

```
"pool-1-thread-1" #11 prio=5 ... waiting on java.util.concurrent.LinkedBlockingQueue@4e50df2e
"pool-1-thread-2" #12 prio=5 ... waiting on java.util.concurrent.LinkedBlockingQueue@4e50df2e
...
"pool-1-thread-50" #60 prio=5 ... waiting on java.util.concurrent.LinkedBlockingQueue@4e50df2e
```

50个线程全部WAITING在`LinkedBlockingQueue`上等待任务——但队列里已经堆积了**30万个任务**。

根因很简单：同事用了 `Executors.newFixedThreadPool(50)`，底层是个无界队列。高峰期任务涌入速度远超处理速度，任务在内存里无限堆积，最终触发OOM。

**线程池的7个参数，配错一个就可能导致线上故障。** 本文逐一拆解每个参数的底层原理和踩坑场景。

2. 无界队列为何是线上OOM的头号杀手
3. 四种拒绝策略的选型决策树
4. 线程池监控指标的正确采集姿势

## 二、七个参数逐一拆解

### 2.1 参数全景

```java
public ThreadPoolExecutor(
    int corePoolSize,                  // 参数1: 核心线程数
    int maximumPoolSize,               // 参数2: 最大线程数
    long keepAliveTime,                // 参数3: 空闲线程存活时间
    TimeUnit unit,                     // 参数4: 时间单位
    BlockingQueue<Runnable> workQueue, // 参数5: 工作队列
    ThreadFactory threadFactory,       // 参数6: 线程工厂
    RejectedExecutionHandler handler   // 参数7: 拒绝策略
)
```

这7个参数决定了线程池的全部行为。下面逐一深入。

### 2.2 参数1&2：corePoolSize 与 maximumPoolSize

#### 核心规则

线程池处理任务的优先级：

```
提交任务
  │
  ├─ 当前线程数 < corePoolSize?
  │     └─ YES → 创建新线程执行（即使有空闲核心线程也不会偷懒）
  │
  ├─ 工作队列未满?
  │     └─ YES → 将任务加入队列
  │
  ├─ 当前线程数 < maximumPoolSize?
  │     └─ YES → 创建新线程执行
  │
  └─ 否则 → 执行拒绝策略
```

**关键点**：只有队列满了才会创建超过 corePoolSize 的线程！这是最容易误解的地方。

#### 场景还原

假设配置：`core=2, max=4, queueCapacity=3`，提交8个任务：

| 序号 | 操作 | poolSize | queueSize | 说明 |
|------|------|----------|-----------|------|
| 任务1 | 直接创建线程 | 1 | 0 | `< core(2)，创建新线程` |
| 任务2 | 直接创建线程 | 2 | 0 | `< core(2)，创建新线程` |
| 任务3 | 加入队列 | 2 | 1 | `core已满，队列未满` |
| 任务4 | 加入队列 | 2 | 2 | `core已满，队列未满` |
| 任务5 | 加入队列 | 2 | 3 | `队列满` |
| 任务6 | 创建额外线程 | 3 | 3 | `队列满 && < max(4)` |
| 任务7 | 创建额外线程 | 4 | 3 | `队列满 && < max(4)` |
| 任务8 | 拒绝 | 4 | 3 | `线程=4=max && 队列满` |

#### 常见踩坑

**坑1：core=0, max=10, SynchronousQueue**

```java
// 所有线程都是"临时工"，空闲即回收
new ThreadPoolExecutor(0, 10, 60L, TimeUnit.SECONDS,
    new SynchronousQueue<>());
```

适用场景：任务量波动极大，如秒杀活动，任务峰谷差异显著。CachedThreadPool 就是这个配置。

**坑2：core=max=10, LinkedBlockingQueue（无界）**

```java
// 线程数永远不超过10，任务无限排队
new ThreadPoolExecutor(10, 10, 0L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>());
```

陷阱：maxPoolSize 形同虚设，队列永远不会满，永远不会创建额外线程，任务无限堆积。

### 2.3 参数3&4：keepAliveTime 与 TimeUnit

`keepAliveTime` 定义了超过 corePoolSize 的线程在空闲多久后被回收。

```java
// 额外线程空闲30秒后销毁
new ThreadPoolExecutor(2, 5, 30L, TimeUnit.SECONDS, workQueue);
```

**关键特性**：
- 仅对超过 corePoolSize 的线程生效
- 核心线程默认永不回收，除非设置 `allowCoreThreadTimeOut(true)`
- JDK 提供的时间单位：NANOSECONDS / MICROSECONDS / MILLISECONDS / SECONDS / MINUTES / HOURS / DAYS

#### 实际案例

```java
// 核心线程2个，最大5个，额外线程空闲2秒后回收
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    2, 5, 2, TimeUnit.SECONDS, new ArrayBlockingQueue<>(10));

// 一波流量过去后，5秒内线程数从5降回2
```

### 2.4 参数5：workQueue（最危险的参数）

这是线上故障的**第一高发参数**。三种常见选择：

#### 类型A：SynchronousQueue（直接交接）

```java
new SynchronousQueue<>()  // 无容量，不存储，直接提交给线程
```

| 特性 | 说明 |
|------|------|
| 容量 | 0（不存储任务） |
| 行为 | 每个插入操作必须等待另一个线程的移除操作 |
| 适用 | 瞬时高并发、任务执行快（Netty IO线程池即用此模式） |
| 风险 | 如果没有空闲线程，立即创建新线程或触发拒绝 |

#### 类型B：LinkedBlockingQueue（无界队列——隐形杀手）

```java
new LinkedBlockingQueue<>()  // 默认容量 Integer.MAX_VALUE (≈21亿)
new LinkedBlockingQueue<>(1000)  // 有界，推荐
```

**为什么是隐形杀手？**

因为`Executors.newFixedThreadPool()`和`Executors.newSingleThreadExecutor()`底层都用无界LinkedBlockingQueue：

```java
// Executors.newFixedThreadPool(10) 实际实现：
return new ThreadPoolExecutor(10, 10, 0L, TimeUnit.MILLISECONDS,
    new LinkedBlockingQueue<Runnable>());  // ← 无界！
```

后果：
1. 队列永远不会满 → `maxPoolSize`形同虚设
2. 任务无限堆积 → 内存持续增长 → Full GC → OOM
3. 请求积压 → 上游超时重试 → 雪崩

#### 类型C：ArrayBlockingQueue（有界队列——生产首选）

```java
new ArrayBlockingQueue<>(1000)  // 固定容量，满了触发拒绝
```

**推荐配置**：生产环境必须用有界队列，容量根据业务QPS和处理耗时计算：

```
队列容量 = 峰值QPS × 单任务平均耗时(秒) × 缓冲系数(1.5~2.0)
```

例如：QPS=100，耗时=0.5s → 50个任务在队列中。建议队列容量 = 50 × 2 = 100。

#### 《阿里巴巴Java开发手册》强制规定

> **【强制】线程池不允许使用 Executors 去创建，而是通过 ThreadPoolExecutor 的方式，这样的处理方式让写的同学更加明确线程池的运行规则，规避资源耗尽的风险。**

说明：Executors 返回的线程池对象的弊端如下：
- FixedThreadPool 和 SingleThreadPool：允许的请求队列长度为 Integer.MAX_VALUE，可能堆积大量请求导致 OOM
- CachedThreadPool 和 ScheduledThreadPool：允许创建线程数量为 Integer.MAX_VALUE，可能创建大量线程导致 OOM

### 2.5 参数6：threadFactory（被低估的参数）

很多开发者传入 `Executors.defaultThreadFactory()` 就不管了，但这在生产环境是**灾难性的**。

#### 为什么必须自定义？

```java
// 这是默认的线程命名：
// pool-1-thread-1, pool-2-thread-3 ...
// jstack 里看到这种名字，你能知道是哪个模块的线程吗？

// 这是自定义命名：
// order-processor-1, payment-handler-3, sms-sender-2
// 瞬间定位问题模块
```

#### 最佳实践：三要素缺一不可

```java
public class NamedThreadFactory implements ThreadFactory {
    private final AtomicInteger counter = new AtomicInteger(1);
    private final String prefix;

    public NamedThreadFactory(String prefix) {
        this.prefix = prefix;
    }

    @Override
    public Thread newThread(Runnable r) {
        Thread t = new Thread(r, prefix + "-" + counter.getAndIncrement());
        t.setDaemon(false);   // ① 非守护线程（防止JVM意外退出）
        t.setUncaughtExceptionHandler((th, ex) ->
            log.error("[{}] 未捕获异常", th.getName(), ex)); // ② 异常处理器
        return t;
    }
}
```

**三要素**：
1. **有意义的名字**：jstack 排查时一眼看出业务模块
2. **非守护线程**：线程池线程应该是普通线程，否则JVM可能意外退出
3. **UncaughtExceptionHandler**：未捕获异常不至于让线程"静默死掉"

### 2.6 参数7：handler（拒绝策略——最后一道防线）

当线程池和队列都满时，新任务触发拒绝策略。JDK 内置四种：

| 策略 | 行为 | 适用场景 | 风险 |
|------|------|----------|------|
| **AbortPolicy**（默认） | 抛出 RejectedExecutionException | 必须感知到任务丢失 | 调用方需捕获异常 |
| **CallerRunsPolicy** | 由调用者线程执行任务 | 流量削峰/反压 | 可能阻塞调用线程 |
| **DiscardPolicy** | 静默丢弃 | 允许丢失的非核心任务 | 任务丢失无感知 |
| **DiscardOldestPolicy** | 丢弃队列头部最老任务 | 优先处理最新数据 | 老数据丢失 |

#### 选型决策树

```
你的任务是否可丢失？
  ├─ 不可丢失
  │     └─ 调用方能承受反压？
  │           ├─ 能 → CallerRunsPolicy
  │           └─ 不能 → AbortPolicy + 调用方重试
  │
  └─ 可丢失
        ├─ 优先处理最新 → DiscardOldestPolicy
        └─ 无所谓新旧 → DiscardPolicy
```

#### 实战建议

90%的场景推荐 `CallerRunsPolicy`，因为它实现了天然的**反压机制**：当线程池处理不过来时，让调用者线程帮忙执行，既减缓了任务提交速度，又不丢任务。

## 四、常见面试题

### Q1：ThreadPoolExecutor 7个参数分别是什么？

**答**：
1. **corePoolSize**：核心线程数，线程池常驻线程数量
2. **maximumPoolSize**：最大线程数，线程池能容纳的最大线程数
3. **keepAliveTime**：空闲线程存活时间，超过 core 的线程空闲超时后回收
4. **unit**：keepAliveTime 的时间单位
5. **workQueue**：工作队列，用于存放等待执行的任务
6. **threadFactory**：线程工厂，用于创建新线程（命名、守护、异常处理）
7. **handler**：拒绝策略，当线程池和队列都满时的处理逻辑

### Q2：线程池的工作流程是怎样的？

**答**：

1. 新任务到来，如果当前线程数 < corePoolSize，**直接创建新线程**执行
2. 如果当前线程数 >= corePoolSize，尝试将任务放入 workQueue
3. 如果 workQueue 已满，且当前线程数 < maximumPoolSize，创建新线程执行
4. 如果 workQueue 已满，且当前线程数 >= maximumPoolSize，执行拒绝策略

**关键点**：只有队列满了才会创建超过 core 的线程。

### Q3：为什么不推荐使用 Executors 创建线程池？

**答**：《阿里巴巴Java开发手册》明确规定：

- **FixedThreadPool / SingleThreadPool**：使用无界 LinkedBlockingQueue（容量 Integer.MAX_VALUE），任务无限堆积可能导致 OOM
- **CachedThreadPool / ScheduledThreadPool**：允许创建 Integer.MAX_VALUE 个线程，可能导致大量线程耗尽系统资源

正确做法：直接使用 `ThreadPoolExecutor` 构造方法，明确指定有界队列和拒绝策略。

### Q4：四种拒绝策略分别适用于什么场景？

**答**：

| 策略 | 场景 |
|------|------|
| AbortPolicy | 关键任务，必须感知到失败，由上层捕获异常处理 |
| CallerRunsPolicy | 需要反压机制，让调用者分担执行实现流量削峰 |
| DiscardPolicy | 可丢弃的非核心任务（如日志、统计） |
| DiscardOldestPolicy | 实时性要求高，优先处理最新数据 |

### Q5：如何合理设置线程池大小？

**答**：分场景计算：

- **CPU密集型**：线程数 = CPU核心数 + 1
- **IO密集型**：线程数 = CPU核心数 × 2（或 × (1 + 平均等待时间/平均计算时间)）
- **混合型**：拆分为两个线程池分别处理

最终值必须通过**压测**验证，公式只是起点。

### Q6：核心线程会被回收吗？

**答**：默认不会。但如果调用 `executor.allowCoreThreadTimeOut(true)`，核心线程在空闲超过 keepAliveTime 后也会被回收。

## 五、总结

1. **7个参数环环相扣**：核心线程、最大线程、超时回收、队列类型、线程工厂、拒绝策略，任意一个参数配置不当都可能导致线上故障
2. **无界队列是 OOM 头号杀手**：`Executors.newFixedThreadPool()` 和 `new LinkedBlockingQueue<>()` 无参构造都会导致任务无限堆积
3. **线程命名不是可有可无**：jstack 排查问题时，有意义的线程名能让你在几十个线程中一眼定位
4. **CallerRunsPolicy 是最推荐的拒绝策略**：天然的流控反压机制，不丢任务
5. **监控先行**：核心指标（poolSize、activeCount、queueSize、completedTaskCount）必须接入监控系统，告警阈值提前设定

**配置口诀**：

> 核心最大管线程，存活时间控回收；
> 队列有界防 OOM，工厂命名助排查；
> 拒绝策略要慎重，生产监控不能少。

**学习建议**：把 Demo 中每个场景的代码跑一遍，手动修改参数值观察行为变化。当你能在面试中画出线程池的工作流程图，并说出每种队列和拒绝策略的适用场景时，你已经超越了 80% 的候选人。
